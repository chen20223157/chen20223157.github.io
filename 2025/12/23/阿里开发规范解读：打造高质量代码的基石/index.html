阿里开发规范解读：打造高质量代码的基石

在当今软件开发的快节奏环境中，代码质量的重要性愈发凸显。阿里巴巴作为全球知名的科技企业，其制定的开发规范在业界具有广泛的影响力。本文将深入探讨阿里开发规范，涵盖其官网地址以及各个重要方面的详细内容。

阿里开发规范官网

阿里巴巴开发规范的官方网站为GitHub - alibaba/p3c: Alibaba Java Coding Guidelines pmd implements and IDE plugin 。在这个网站上，开发者可以获取到最新版本的开发手册，包括 Java、Android 等多种语言和平台的规范文档。同时，官网还提供了丰富的资源，如插件下载、示例代码、社区讨论等，方便开发者深入学习和交流。

Java 开发规范

命名风格

  1. 强制规则
    • 所有编程相关的命名均不能以下划线或美元符号开始，也不能以下划线或美元符号结束。例如，_name、name_、$Object 等命名方式都是不允许的。
    • 严禁使用拼音与英文混合的方式，更不允许直接使用中文命名。正确的英文拼写和语法能极大提升代码的可读性，避免歧义。像 ali、alibaba、taobao 等国际通用名称是符合规范的，而 DaZhePromotion（打折）、getPingfenByName()（评分）等则不符合要求。
    • 代码和注释中要严格避免使用任何人类语言中的种族歧视性或侮辱性词语。例如，应避免使用 blackList、whiteList、slave、SB、WTF 等词汇，可采用 blockList、allowList、secondary 等替代。
    • 类名使用 UpperCamelCase 风格，但 DO / PO / DTO / BO / VO / UID 等除外。例如，ForceCode、UserDO、HtmlDTO 是正确的，而 forcecode、UserDo 则错误。
    • 方法名、参数名、成员变量、局部变量统一使用 lowerCamelCase 风格，如 localValue、getHttpMessage()、inputUserId。
    • 常量命名全部大写，单词间用下划线隔开，语义要完整清楚。例如，MAX_COUNT、DEFAULT_TIMEOUT。
    • 抽象类命名使用 Abstract 或 Base 开头；异常类命名使用 Exception 结尾；测试类命名以要测试的类的名称开始，以 Test 结尾。
    • 类型与中括号紧挨相连来定义数组，如 int[] arrayDemo，而不是 int arrayDemo[]。
    • POJO 类中的布尔类型变量，不要加 is 前缀，否则部分框架解析可能会引起序列化错误。比如，定义为 Boolean isDeleted 的属性，其方法也是 isDeleted()，框架反向解析时可能会误以为对应的属性名称是 deleted，导致属性获取不到并抛出异常。
    • 包名统一使用小写，点分隔符之间有且仅有一个自然语义的英语单词，且统一使用单数形式，类名若有复数含义可使用复数形式。例如，应用工具类包名为 com.alibaba.ei.kunlun.aap.util，类名为 MessageUtils。
    • 避免在子父类的成员变量之间、不同代码块的局部变量之间采用完全相同的命名，以免降低代码的可理解性。
    • 杜绝完全不规范的英文缩写，避免望文不知义。例如，不能将 AbstractClass 缩写成 AbsClass，将 condition 缩写成 condi。
  2. 推荐规则
    • 为了使代码自解释，自定义编程元素命名时应使用完整的单词组合来表达。例如，JDK 中对某个对象引用的 volatile 字段进行原子更新的类名为 AtomicReferenceFieldUpdater。
    • 在常量与变量命名时，表示类型的名词放在词尾，提升辨识度。例如，userList、orderId。
    • 如果模块、接口、类、方法使用了设计模式，命名时应体现出具体模式，便于阅读者快速理解架构设计思想。比如，OrderFactory（工厂模式）、ResourceObserver（观察者模式）。
    • 接口类中的方法和属性不要加任何修饰符号（public 也不要加），保持代码简洁性，并加上有效的 Javadoc 注释。尽量不在接口里定义常量，若要定义，需确保该常量与接口方法相关且是整个应用的基础常量。
    • 对于 Service 和 DAO 类，基于 SOA 理念，暴露出来的服务必须是接口，内部实现类用 Impl 后缀与接口区别，如 CacheServiceImpl 实现 CacheService 接口。若是形容能力的接口名称，可取对应的形容词（通常是 - able 结尾）为接口名，如 AbstractTranslator 实现 Translatable 接口。
  3. 参考规则
    • 枚举类名带上 Enum 后缀，枚举成员名称全大写，单词间用下划线隔开。例如，ProcessStatusEnum 的成员名称 SUCCESS、UNKNOWN_REASON。
    • Service / DAO 层方法命名有特定规约：获取单个对象的方法用 get 做前缀；获取多个对象的方法用 list 做前缀，复数结尾，如 listObjects；获取统计值的方法用 count 做前缀；插入的方法用 save /insert 做前缀；删除的方法用 remove /delete 做前缀；修改的方法用 update 做前缀。领域模型命名也有规约，数据对象为 xxxDO（xxx 为数据表名），数据传输对象为 xxxDTO（xxx 为业务领域相关名称），展示对象为 xxxVO（xxx 一般为网页名称），POJO 是 DO / DTO / BO / VO 的统称，禁止命名成 xxxPOJO。

常量定义

  1. 强制规则
    • 不允许任何魔法值（未经定义的常量）直接出现在代码中。例如，不能在代码中直接写 if (status == 1)，而应先定义常量 public static final int STATUS_SUCCESS = 1;，然后使用 if (status == STATUS_SUCCESS)。
    • 不要使用一个常量类维护所有常量，应按常量功能进行归类，分开维护。比如，缓存相关常量放在类 CacheConsts 下，系统配置相关常量放在类 ConfigConsts 下。
  2. 推荐规则
    • 常量的复用层次有区别时，应分开定义成不同层次的常量，避免低层次常量被高层次复用，造成常量污染。例如，PageInfo 类中的 PAGE_SIZE 常量，不同业务模块可能有不同的分页需求，应根据业务模块的层次分别定义。
    • 如果变量值仅在一个范围内变化，且带有名称之外的延伸属性，应定义为枚举类型。例如，订单状态可定义为枚举类型 public enum OrderStatus { PENDING, PAID, SHIPPED, COMPLETED }。

  1. 强制规则
    • 如果大括号内为空，简洁写成 {} 即可，中间无需换行和空格；非空代码块时，左大括号前不换行，左小括号和右边相邻字符之间不出现空格，右小括号和左边相邻字符之间也不出现空格，左大括号前需要加空格。例如：

java

  • 任何二目、三目运算符的左右两边都需要加一个空格，包括赋值运算符 =、逻辑运算符 &&、加减乘除符号等。例如，int result = a + b;。
  • 采用 4 个空格缩进，禁止使用 tab 字符。在 IDEA 中设置 tab 为 4 个空格时，请勿勾选 Use tab character；在 eclipse 中，必须勾选 insert spaces for tabs。
  • 在进行类型强制转换时，右括号与强制转换值之间不需要任何空格隔开。例如，int num = (int) value;。
  • 单行字符数限制不超过 120 个，超出需要换行，换行时遵循以下原则：第二行相对第一行缩进 4 个空格，从第三行开始不再继续缩进；运算符与下文一起换行；方法调用时，多个参数需要换行时，在逗号后进行；在括号前不要换行。例如：

java

  • 方法参数在定义和传入时，多个参数逗号后边必须加空格。例如，public void method(int param1, int param2) {}。
  • IDE 的 text file encoding 设置为 UTF - 8，文件的换行符使用 Unix 格式，不要使用 Windows 格式。
  1. 推荐规则
    • 单个方法的总行数不超过 80 行，包括方法签名、结束右大括号、方法内代码、注释、空行、回车及任何不可见字符。将个性和共性逻辑分开，提取成额外方法，使主干代码更清晰，便于复用和维护。
    • 不同逻辑、不同语义、不同业务的代码之间插入一个空行分隔开来，提升代码可读性，但不要插入多个空行。

OOP 规约

  1. 强制规则
    • 避免通过一个类的对象引用访问此类的静态变量或静态方法，直接用类名访问即可，无谓增加编译器解析成本。例如，应使用 ClassName.staticMethod()，而不是 new ClassName().staticMethod()。
    • 所有的覆写方法，必须加 @Override 注解，明确标识该方法是对父类方法的重写，提高代码的可读性和可维护性。
    • 相同参数类型、相同业务含义，才可以使用 Java 的可变参数，避免使用 Object。可变参数必须放置在参数列表的最后。例如：

java

  • 外部正在调用或者二方库依赖的接口，不允许修改方法签名，避免对接口调用方产生影响。接口过时必须加 @Deprecated 注解，并清晰说明采用的新接口或者新服务。
  • 不能使用过时的类或方法。若使用，应及时查找新的实现方式。例如，java.util.Date 的部分方法已过时，应使用 java.time.LocalDate 等新的日期时间类。
  • Object 的 equals 方法容易抛空指针异常，应使用常量或确定有值的对象来调用 equals。例如，"string".equals(str)，而不是 str.equals("string")。
  • 所有相同类型的包装类对象之间值的比较，全部使用 equals 方法比较。因为在 -128 至 127 范围内的赋值，Integer 对象是在 IntegerCache.cache 产生，会复用已有对象，此区间内的 Integer 值可直接使用 == 判断，但区间之外的所有数据，都会在堆上产生，不会复用已有对象，所以推荐使用 equals 方法判断。
  • 所有的 POJO 类属性必须使用包装数据类型；RPC 方法的返回值和参数必须使用包装数据类型；所有的局部变量推荐使用基本数据类型。POJO 类属性没有初值可提醒使用者显式赋值，避免 NPE 问题。例如，数据库查询结果可能为 null，用基本数据类型接收有 NPE 风险。而包装数据类型的 null 值能表示额外信息，如远程调用失败、异常退出。
  • 定义 DO / DTO / VO 等 POJO 类时，不要设定任何属性默认值，避免数据提取时未置入具体值，更新其他字段时附带更新此字段，导致数据错误。
  • 序列化类新增属性时，若要保持兼容性，不要修改 serialVersionUID 字段，避免反序列失败；若完全不兼容升级，为避免反序列化混乱，应修改 serialVersionUID 值，注意 serialVersionUID 不一致会抛出序列化运行时异常。
  • 构造方法里面禁止加入任何业务逻辑，如果有初始化逻辑，请放在 init 方法中。
  • POJO 类必须写 toString 方法，方便在方法执行抛出异常时，直接调用 POJO 的 toString() 方法打印其属性值，排查问题。使用 IDE 工具生成 toString 方法时，如果继承了另一个 POJO 类，注意在前面加 super.toString()。
  1. 推荐规则
    • 当一个类有多个构造方法，或者多个同名方法，这些方法应按顺序放置在一起，类内方法定义顺序依次是：公有方法或保护方法 > 私有方法 > getter/setter 方法。
    • 循环体内，字符串的连接方式，使用 StringBuilder 的 append 方法进行扩展，避免每次循环都创建新的 StringBuilder 对象，造成内存资源浪费。例如：

java

  • 类成员与方法访问控制从严：如果不允许外部直接通过 new 来创建对象，构造方法必须是 private；工具类不允许有 public 或 default 构造方法；类非 static 成员变量并且与子类共享，必须是 protected；类非 static 成员变量并且仅在本类使用，必须是 private；类 static 成员变量如果仅在本类使用，必须是 private；若是 static 成员变量，必须考虑是否为 final；类成员方法只供类内部调用，必须是 private；类成员方法只对继承类公开，限制为 protected。严格控制访问范围，有利于模块解耦。

集合处理

  1. 强制规则
    • 关于 equals 方法和 hashCode 方法的处理，只要重写 equals，就必须重写 hashCode。因为 equals 方法用于判断两个对象是否相等，而 hashCode 方法用于在哈希表等集合中确定对象的存储位置，如果两个对象 equals 方法返回 true，但 hashCode 方法返回不同值，会导致在集合中出现逻辑错误。
    • 集合初始化时，要指定合理的初始容量。例如，ArrayList 若已知大概元素数量，应在初始化时指定容量，避免多次扩容带来的性能开销。ArrayList list = new ArrayList(100);
    • 使用 entrySet 遍历 Map 类集合，而不是 keySet。因为使用 keySet 遍历会先获取键集，再通过键获取值，增加了一次查找操作，而 entrySet 直接返回键值对集合，效率更高。例如：

java

  • 在使用 HashMap 和 HashSet 时，要注意其线程不安全特性。如果在多线程环境下使用，应考虑使用 ConcurrentHashMap 和 CopyOnWriteArraySet 等线程安全的集合类。
  • 不要在 foreach 循环里进行元素的 remove / add 操作。foreach 循环是基于迭代器实现的，在循环中进行元素的增删操作会导致迭代器状态不一致，抛出 ConcurrentModificationException。如果需要删除元素，可使用迭代器的 remove 方法；如果需要添加元素，可考虑使用其他数据结构暂存，循环结束后再进行添加。
  1. 推荐规则
    • 集合转数组时，使用 Collection 的 toArray(T[] a) 方法，并传入类型完全一样的数组，大小就是 list.size()。例如：

java

  • 使用工具类 Arrays 或 Collections 提供的方法进行集合操作，如 Arrays.asList()、Collections.sort() 等，这些方法经过优化，代码更简洁高效。
  • 对于频繁进行插入和删除操作的场景，优先使用 LinkedList 而不是 ArrayList，因为 LinkedList 的插入和删除操作时间复杂度为 O (1)，而 ArrayList 为 O (n)。
